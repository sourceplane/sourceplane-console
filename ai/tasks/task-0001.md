# Required Outcomes

1. Add a Supabase/Postgres-compatible identity schema migration.
   - Use an identity-owned schema or unambiguous identity table namespace.
   - Preserve all existing identity facts: users, login challenges, sessions, service principals, API keys, and identity event outbox.
   - Preserve indexes and uniqueness guarantees equivalent to the current D1 migration.
   - Use Postgres-appropriate types and constraints. Avoid cross-context foreign keys.

2. Add a production repository adapter for identity.
   - Implement the existing `IdentityRepository` contract for Postgres.
   - Keep SQL and driver details inside the adapter layer.
   - Use parameterized queries only.
   - Ensure multi-record identity mutations that must be atomic are transaction-safe in the Postgres path.
   - Preserve token hash, API-key, session, and event-outbox semantics.

3. Wire the identity Worker to choose the correct repository.
   - Keep D1 usable for current tests and local development.
   - Add a Hyperdrive/Postgres path using the existing `sourceplane-db` Hyperdrive binding/resource.
   - Fail clearly in production if no production database binding is available.
   - Do not require a live Supabase database for the existing test suite.
   - Use `wrangler` to inspect the configured Hyperdrive path and, if needed for local verification, generate temporary database credentials without committing or logging them in full.

4. Update docs and configuration carefully.
   - Update `apps/identity-worker/README.md` to explain D1 local/test compatibility and the production Supabase/Postgres + Hyperdrive path.
   - Do not commit raw Supabase credentials or connection strings.
   - Do not invent Tactonic, Terraform, Supabase project, Cloudflare account, or secondary Hyperdrive names.
   - If Wrangler configuration changes create or update Cloudflare resources, verify the resulting resource state with `wrangler` or the Cloudflare API and include the verification result in the report.

5. Add focused tests.
   - Existing D1 identity tests must keep passing.
   - Add tests for the new repository selection behavior.
   - Add repository-level coverage for the Postgres adapter where possible without network access. If a full Postgres execution test is not practical in this repo yet, add the strongest offline coverage available and record the remaining gap in the report.
